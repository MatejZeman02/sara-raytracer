00 · Overview

Why build a custom LUT?

Most display pipelines use off-the-shelf ACES or a simple Reinhard curve. This one is hand-tuned: it borrows ideas from ACES 2.0, adds a film-like emulsion simulation, and does gamut compression in a perceptually uniform space. The result is baked into a 3D lookup table so it runs at zero cost at render time.

01 · Input Grid

20 stops of HDR scene in a unit cube

The LUT maps a 3D unit cube [0,1]³ to display-ready colors. But before anything else, we need to decide: what does position 0 mean, and what does position 1 mean, in terms of actual light?

We use a log scale: position 0 on each axis is −10 EV, position 1 is +10 EV. That's 20 stops — enough to cover everything from a pitch-black cave to the surface of the sun in direct light.

This is critical. A linear grid would waste most of its resolution on very bright values, where the human eye can barely tell the difference. A log grid packs resolution where you actually need it: in the midtones and shadows.

32³ = 32,768 entries — each holding an RGB triplet. The full LUT is about 384KB as float32. Coarse enough to compute offline, fine enough that trilinear interpolation at sample time is seamless.
Formula
# grid [0,1] → EV stops → linear stops = grid_01 * (20.0) + (-10.0) linear_acescg = 2.0 ** stops

02 · Colorspace Transform

ACEScg to linear sRGB

ACEScg is the working space used in VFX and rendering. It's bigger than sRGB — it can represent colors that no display can show. The first step is a matrix transform to get into the sRGB primaries, using CAT02 chromatic adaptation to handle the different white points.

Think of this as rotating the color space. ACEScg has a very wide gamut — it was designed to capture everything a camera or renderer could produce. sRGB is the gamut of a consumer monitor. The matrix remaps one set of primaries to the other.

After this step, many colors will be out of gamut — meaning one or more channels will go negative. That's not an error, it's expected — those are colors that physically can't be shown on a display. We'll deal with them in the next step.

CAT02 is the chromatic adaptation transform from CIECAM02. It adapts D60 (ACES white point) to D65 (sRGB white point), so whites look neutral on a standard monitor rather than slightly warm.

03 · Asymmetric Crosstalk

Film emulsion bleed

This is the step that makes the whole thing feel like film rather than a digital grade. Real film emulsions aren't perfectly isolated — the dye layers bleed into each other. This matrix simulates that, deliberately and asymmetrically.

The bleed is intentionally one-directional: red bleeds into green, green into blue, and blue into red. This creates a characteristic "warmth rotation" — a red sunset picks up a slightly golden-green cast, blues take on a subtle violet note.

Each channel keeps 85% of itself and donates 5% and 10% to the other two. The asymmetry is what makes it look analog rather than like a flat de-saturate.

04 · Gamut Compression

Pulling out-of-gamut colors back in

After the colorspace transform and crosstalk, many colors will have negative RGB values — they're outside what sRGB can represent. We need to compress them back in. The trick is doing this in a perceptually uniform space so the compression looks natural.

The approach here is elegant: convert to Oklab, then do a binary search on the chroma (a and b axes). We're asking: "how much can we scale down the chroma while keeping lightness exactly the same, until no channel goes negative?"

Fifteen iterations of bisection gives us about 1/32768th precision — more than enough for float32. The key insight is we stay in Oklab the whole time, so we're compressing perceptually, not mathematically. Hue and lightness are preserved; only saturation is reduced.

Why Oklab? Oklab is designed so that equal distances correspond to equal perceived color differences. Compressing chroma in Oklab produces a much cleaner result than clipping channels directly — no hue shifts, no weird "neon" artifacts.
# 15-iteration binary search on chroma scale for _ in range(15): mid = (low + high) / 2.0 oklab_test[..., 1:3] *= mid # scale a,b rgb_test = oklab_to_rgb(oklab_test) high = where(rgb_test < 0, mid, high) low = where(rgb_test >= 0, mid, low)

05 · Highlight Desaturation

Fading to white in the highlights

Very bright areas in real scenes tend to blow out to white — film handles this gracefully by desaturating highlights before they clip. This step simulates that: as Oklab lightness rises toward pure white, chroma is gently rolled off to zero.

The rolloff starts at L = 1.5 and reaches zero at L = 2.0. Those are Oklab lightness values, where 1.0 is "white" on a display — so this is working in the super-white, specular highlight range.

Critically, a Hermite smoothstep is used instead of a linear ramp. This gives a photographic S-curve rolloff — the transition starts gentle, accelerates, then eases back to zero. Linear rolloffs tend to look mechanical.

This is still done in Oklab, so the desaturation is perceptually uniform. A saturated red highlight will rotate toward a neutral white through perceptually similar intermediate colors — not toward a washed-out pink.

06 · Uchimura Tonemapping

The Uchimura curve

Now we need to compress the full HDR range into [0,1] for display. The Uchimura operator is a piecewise curve — toe, linear section, shoulder — originally designed for game engines but well-suited here because its parameters map directly to photographic concepts.

Crucially, tonemapping is applied to the luminance norm — the maximum channel — not to each channel independently. After computing the scale factor, we multiply all three channels by the same number. This preserves hue perfectly, which independent per-channel tonemapping does not.

07 · Output

The final γ encode and what we get

After tonemapping, values are in linear light [0,1]. One last step: apply a 2.2 power encode. This gamma bake converts the linear signal to a display-referred encoding, which is what monitors and image files expect.

Note: this is a pure power curve (x^2.2), not the full sRGB transfer function with its linear segment at the toe. For a 3D LUT, the difference is negligible — the linear segment only matters at very low values, and the LUT's interpolation handles that region fine.

The output is saved as a float32 NumPy array shaped [32, 32, 32, 3]. This is deliberately kept as raw data rather than a .cube or .3dl file — it's designed for Numba-accelerated sampling at render time, where direct array indexing is faster than parsing an ASCII LUT format.
